# Flyway Migrations

Safe schema-evolution practices for Spring Boot + Flyway. Adapted from ECC's
`database-migrations` and `postgres-patterns` skills.

## Before generating

If not already known from the conversation, confirm:

- Target database (PostgreSQL, MySQL, …) — DDL syntax and types differ.
- Migration tool: this skill assumes **Flyway**. If the project uses Liquibase instead,
  say so and adapt (Liquibase uses changelog XML/YAML/JSON, not `V__` SQL files).

## File naming and location

- Place versioned migrations in `src/main/resources/db/migration`.
- Name them `V<version>__<description>.sql` — **two** underscores after the version:
  - `V1__create_note_table.sql`
  - `V2__add_index_on_note_title.sql`
- Versions are ordered; never renumber or edit a migration that has already run anywhere.
  To change something, write a **new** migration.
- Use `R__<description>.sql` only for genuinely repeatable objects (views, functions).

## Forward-only, immutable discipline

- Migrations that have been applied are **immutable**. Editing one breaks Flyway's
  checksum validation and corrupts the migration history across environments.
- Every change is additive going forward. Need to undo something? Add a new migration that
  reverses it.
- Keep one logical change per migration where practical — easier to review and to revert.

## Safe DDL patterns (avoid downtime and data loss)

Treat these as the default playbook:

- **Adding a column:** add it nullable, or with a default. Adding a `NOT NULL` column
  without a default to a populated table fails or locks the table.
  - Zero-downtime pattern: (1) add nullable column, (2) backfill in batches, (3) add the
    `NOT NULL` constraint in a later migration once data is populated.
- **Dropping a column/table:** stop writing to it in the application first, ship that,
  *then* drop it in a later migration. Never drop in the same release that still references
  it.
- **Renaming:** prefer add-new + backfill + switch reads/writes + drop-old over an
  in-place rename, which is a breaking change for any running old code.
- **Indexes on large tables (PostgreSQL):** use `CREATE INDEX CONCURRENTLY` to avoid
  locking writes. Note it cannot run inside a transaction, so keep it in its own migration
  and configure Flyway accordingly.
- Always add indexes for foreign keys and for columns used in `WHERE`/`ORDER BY`.

## Proper data types

- Timestamps: `timestamptz` (PostgreSQL) for instants; avoid bare `timestamp` for
  audit fields.
- Text: `text` for unbounded strings, `varchar(n)` when a real limit exists.
- Money: `numeric`/`decimal`, never floating point.
- Keys: `bigint` (or `bigserial`/identity) for surrogate ids; `uuid` if you use UUIDs.
- Enforce integrity with `NOT NULL`, `UNIQUE`, foreign keys, and `CHECK` constraints —
  the database is the last line of defense.

**Example — `V1__create_note_table.sql` (PostgreSQL):**
```sql
CREATE TABLE note (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title      VARCHAR(255) NOT NULL,
    content    TEXT,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_note_title ON note (lower(title));
```

## Spring Boot wiring

- Add the `flyway-core` dependency (and `flyway-database-postgresql` on newer Flyway with
  PostgreSQL). Flyway runs automatically on startup.
- Let **Flyway own the schema**: set `spring.jpa.hibernate.ddl-auto=validate` so Hibernate
  checks the mapping against the migrated schema but never alters it.
- Keep migrations in version control alongside the code that depends on them.

## Anti-patterns to avoid

- Editing an already-applied migration.
- `ddl-auto=update` together with Flyway (two tools fighting over the schema).
- Adding `NOT NULL` columns without defaults to populated tables in one step.
- Dropping columns/tables in the same release that still uses them.
- Non-concurrent index creation on large, live tables.
- Putting environment-specific data in versioned migrations (use repeatable or seed scripts
  deliberately).
